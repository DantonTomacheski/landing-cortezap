Agendar uma demo de software é como ir a um test drive de carro. O vendedor mostra o melhor ângulo, a estrada mais lisa e o ar-condicionado no máximo. Você sai da concessionária empolgado. Depois de 3 meses com o carro no trânsito da cidade, percebe que o consumo é alto, o banco é duro e a multimídia trava.

Demos de software para barbearia funcionam do mesmo jeito. Interface com design impecável, dados fictícios que sempre funcionam, fluxo que roda sem erro. Tudo pensado para impressionar. O problema é que sua operação real não tem dados perfeitos nem clientes fictícios que sempre seguem o roteiro.

Este guia ensina a assistir demos com olhar de quem vai comprar e usar, não de quem vai aplaudir uma apresentação.

## O problema real

O barbeiro assiste 3 demos em 3 dias. As três pareceram boas. As três tinham interface bonita. As três prometiam resolver tudo. Agora ele está mais confuso do que antes.

Isso acontece porque demos são projetadas para serem parecidas na superfície. Todas mostram agenda, relatórios e agendamento. A diferença está nos detalhes que a demo não mostra: velocidade com dados reais, comportamento quando algo dá errado, experiência do cliente final, suporte depois da venda.

Sem saber o que perguntar e onde olhar, o barbeiro compara aparência em vez de substância. E aparência não paga boleto.

## Erro comum

O primeiro erro é assistir demo sem preparação. O vendedor pergunta "o que você quer ver?" e o barbeiro responde "tudo". Resultado: uma apresentação de 45 minutos que cobre tudo superficialmente e nada profundamente.

O segundo erro é focar na interface. "Achei mais bonito" não é critério de decisão para ferramenta de trabalho. Funcionalidade, velocidade e confiabilidade importam mais que design.

Outro erro é comparar feature por feature numa planilha. "Esse tem relatório ABC, aquele não." Sem contexto, isso é inútil. A pergunta é: você vai usar o relatório ABC? Se não, é irrelevante que um sistema tenha e o outro não.

E o erro mais sutil: acreditar no que o vendedor diz sobre a concorrência. "O sistema X não faz isso, só nós fazemos." Quase sempre é mentira ou meia-verdade. Verifique por conta própria.

## O que funciona na prática

### Antes da demo: defina 3 cenários para testar

Não assista demo passivamente. Prepare 3 situações reais da sua operação e peça para o vendedor demonstrar:

- **Cenário 1 — Agendamento pelo canal principal**: se seus clientes agendam pelo WhatsApp, peça para ver exatamente como funciona. Da mensagem do cliente à confirmação na agenda do barbeiro. Passo a passo.
- **Cenário 2 — Situação de conflito**: dois clientes tentam o mesmo horário. Um barbeiro falta e precisa transferir os agendamentos. Um cliente cancela em cima da hora. Como o sistema lida?
- **Cenário 3 — Relatório financeiro**: peça para ver o relatório de comissão de um barbeiro no mês. Mostra por serviço? Mostra total líquido? É claro o suficiente para o barbeiro entender sem explicação?

Se o vendedor não conseguir demonstrar esses cenários, o sistema provavelmente não os resolve bem.

### Durante a demo: 8 perguntas que revelam a verdade

1. **"Posso ver isso com dados reais, não de demonstração?"** — Dados fictícios sempre funcionam. Dados reais revelam bugs e lentidão.

2. **"Quanto tempo leva o onboarding completo?"** — Se dizem "instantâneo" para uma barbearia com 5 barbeiros e 20 serviços, estão simplificando demais.

3. **"O que acontece quando o WhatsApp cai?"** — Se o sistema depende de WhatsApp, precisa ter plano B. Se o vendedor nunca pensou nisso, o sistema não é maduro.

4. **"Posso falar com um cliente atual que usa há mais de 6 meses?"** — Referência real vale mais que 10 demos. Se negarem, desconfie.

5. **"Qual a funcionalidade que mais reclamam?"** — Vendedor honesto admite os pontos fracos. Vendedor que diz "ninguém reclama" está mentindo.

6. **"Se eu quiser sair, consigo exportar meus dados?"** — A resposta mostra se a empresa confia no produto ou prende o cliente pela dependência.

7. **"Quem é o suporte depois que eu contratar? Você mesmo ou outro time?"** — Se o contato pós-venda é outro, peça para conhecer esse time antes.

8. **"Quanto custa de verdade por mês, somando tudo?"** — Mensalidade + extras + taxas. Peça o número final, não o "a partir de".

### Depois da demo: use o trial como prova real

A demo é a primeira impressão. O trial é o namoro. Não decida pela primeira impressão.

Se o sistema oferece trial gratuito:

- Configure com seus dados reais (serviços, barbeiros, horários)
- Peça para clientes reais agendarem (sem explicar como funciona — o fluxo precisa ser intuitivo)
- Use por pelo menos 5 dias úteis
- Acione o suporte com uma dúvida real e cronometre
- No fim do trial, responda: "Eu pagaria por isso?" Se hesitar, não contrate.

### Como pontuar comparação entre dois ou mais sistemas

Crie uma tabela simples com 5 critérios e note de 1 a 5:

- Resolve meu problema principal? (peso 3)
- Experiência do cliente é boa? (peso 2)
- Suporte respondeu rápido e bem? (peso 2)
- Custo total cabe no orçamento? (peso 2)
- Equipe consegue usar sem dificuldade? (peso 1)

Multiplique nota pelo peso e some. O maior número é o candidato mais forte. Não é ciência exata, mas é melhor que "senti que aquele era melhor".

## Quando isso não resolve

Se os dois ou três sistemas são muito parecidos na pontuação, o diferencial vai ser intangível: confiança no fornecedor, feeling com o suporte, alinhamento de valores. Nesses casos, escolha o que tem menos risco (sem fidelidade, com trial, com suporte acessível).

Se você não tem tempo para assistir demos e fazer trials, peça para um funcionário de confiança fazer a avaliação com os critérios que você definiu. Delegar a pesquisa é melhor do que decidir sem pesquisa.

## Como medir se melhorou

- **A decisão foi feita com critério documentado?** Se você tem uma tabela comparativa ou notas, foi melhor que "achei bonito".
- **Sem surpresas nos primeiros 30 dias?** Se o sistema entrega o que a demo mostrou, a avaliação foi precisa.
- **O sistema resolve o que foi demonstrado nos 3 cenários?** Se sim, você fez as perguntas certas.
- **A equipe adotou sem resistência significativa?** Boa avaliação inclui testar usabilidade com quem vai usar.
